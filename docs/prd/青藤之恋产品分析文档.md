## 一、产品概述

### 1.1 产品定位

青藤之恋是一款面向**高学历优质青年**的严肃婚恋交友平台。slogan：「在青藤之恋，遇见和你一样优秀的TA」。

核心差异化定位：**真实、可靠、温暖、走心**。与探探、Soul 等泛社交/泛娱乐化交友平台不同，青藤之恋强调「学历认证+实名认证」双重门槛，主打高知青年群体的严肃婚恋需求。

### 1.2 目标用户

- **年龄**：以 85后、90后为主，逐步覆盖 00后
- **学历**：本科及以上，硕士/博士比例较高
- **特征**：一线/新一线城市打拼的单身青年，社交圈有限，缺少认识异性朋友的渠道
- **核心诉求**：严肃恋爱/婚姻导向，而非短期社交或娱乐

### 1.3 核心价值主张

| 维度 | 描述 |
|------|------|
| **真实** | 实名认证+学历认证双重审核，人工审核相册和资料 |
| **可靠** | 双向匹配才能聊天，杜绝骚扰 |
| **温暖** | 村口社区、"唠唠"等轻社交功能，营造陪伴感 |
| **走心** | 微信公众号每日推送嘉宾采访、脱单故事，构建情感连接 |

## 二、核心功能模块

### 2.1 用户认证体系

青藤之恋最核心的壁垒之一。

| 认证类型 | 说明 |
|----------|------|
| **实名认证** | 身份证实名验证 |
| **学历认证** | 需提交学历证书/学信网验证，人工审核 |
| **相册审核** | 每次提交头像/相册均需后台人工审核 |
| **资料审核** | 个人资料提交后人工审核 |

> 所有用户必须通过实名和学历双重认证才能进行互动。

### 2.2 匹配推荐 —「寻觅」

核心匹配功能模块，类似 Tinder 的滑动卡片模式：

- **每日推荐**：首页每天智能推荐符合用户条件的嘉宾
- **喜欢（Like）**：对推荐用户点击「喜欢」
- **超级喜欢（Super Like）**：付费功能，表达更强兴趣，对方会收到突出通知
- **超级消息（Super Message）**：付费功能，在匹配前直接发送一条消息给对方
- **双向匹配**：双方互相喜欢才能开启聊天，避免单方骚扰

### 2.3 即时通讯 —「聊天」

- 仅双向匹配后可开启聊天
- 无噪音无打扰的设计理念
- 支持文字、图片等基础聊天功能

### 2.4 社区 —「村口」

轻量级社区功能，类似朋友圈+话题广场：

- **唠唠**：用户发布动态（类似朋友圈/微博），可获得点赞和评论
- 社区氛围温暖，用户可在此分享生活日常、寻求建议
- 增强用户留存和日常活跃度

### 2.5 个人主页

展示用户核心信息：
- 学校（如北京邮电大学）
- 学历（本科/硕士/博士）
- 职业/行业
- 年薪区间
- 所在城市
- 身高
- 星座
- 个人照片

### 2.6 脱单故事

- 真实用户成功案例展示
- 从认识到恋爱、结婚的时间线
- 增强品牌信任感和用户信心

### 2.7 超级曝光

- 付费/限时功能
- 微信公众号每日推送嘉宾采访
- 让用户获得更多曝光机会

### 2.8 线下活动

- 不定期举办线下交友活动
- 增强真实社交场景

## 四、产品交互与设计特色

### 4.1 核心交互流程

```
注册 → 实名认证 → 学历认证 → 完善资料 → 人工审核
       ↓
   每日推荐（寻觅）→ 喜欢/超级喜欢 → 双向匹配 → 聊天
       ↓
   村口社区参与 → 线下活动 → 脱单
```

### 4.2 设计理念

- **严肃交友**：审核机制严格，过滤非严肃用户
- **安全感**：双向匹配才能聊天，女性用户体验友好
- **温暖调性**：品牌色系偏暖，文案走心
- **社区感**：通过「村口」和「脱单故事」构建用户粘性

### 7.1 🎲 「缘分盲盒」— 游戏化运气匹配

**设计理念**：打破严肃推荐的框架，引入**纯运气驱动**的游戏化匹配，降低用户决策疲劳，增加惊喜感和产品趣味性。

#### 功能描述

每天为用户提供一次（或多付费购买额外次数）的随机匹配机会：

| 玩法 | 说明 |
|------|------|
| **缘分盲盒** | 系统随机推送一个符合条件的异性用户（隐藏照片、模糊个人资料），用户点开前完全不知道对方是谁 |
| **缘分骰子** | 用户掷骰子，不同点数对应不同惊喜（1-2点：普通匹配 / 3-4点：同城匹配 / 5点：高匹配度用户 / 6点：获得一次「超级喜欢」免费使用权） |
| **缘分扭蛋** | 消耗「缘分币」抽取匹配卡片，稀有度分 N/R/SR/SSR，SSR 级别为系统算法认定的高度匹配用户 |

#### 详细设计

```
入口：首页「寻觅」Tab 旁新增「缘分」Tab

┌─────────────────────────────────────┐
│          🎲 缘分盲盒               │
│                                     │
│    ┌───────────────┐                │
│    │      ?        │   ← 神秘卡片   │
│    │   点击揭晓     │                │
│    └───────────────┘                │
│                                     │
│   今日剩余次数：1/1                 │
│   [ 开启盲盒 ]   [ +1 次 (¥X) ]    │
│                                     │
│   ┌─ 缘分骰子 ─────────────────┐   │
│   │  🎲 掷骰子获得随机惊喜     │   │
│   │  今日已掷：0/1             │   │
│   └────────────────────────────┘   │
│                                     │
│   ┌─ 缘分扭蛋 ─────────────────┐   │
│   │  🔮 消耗缘分币抽取匹配卡片 │   │
│   │  缘分币余额：💎 × 5        │   │
│   │  [ 单抽 ¥3 ] [ 十连 ¥25 ]   │   │
│   └────────────────────────────┘   │
└─────────────────────────────────────┘
```

#### 规则设计

| 玩法 | 免费次数 | 付费价格 | 限制 |
|------|---------|---------|------|
| **缘分盲盒** | 1次/天 | ¥5/次 | 每日最多5次 |
| **缘分骰子** | 1次/天 | ¥3/次 | 每日最多3次 |
| **缘分扭蛋** | 无 | 单抽 ¥3 / 十连 ¥25 | 无上限 |

#### SSR 概率设计（扭蛋）

| 稀有度 | 概率 | 说明 |
|--------|------|------|
| N (普通) | 60% | 随机异城用户 |
| R (稀有) | 25% | 同城用户 |
| SR (超稀有) | 12% | 同城+高匹配分 |
| SSR (传说) | 3% | 同城+高匹配分+对方也活跃 |

#### 社交传播设计

- 抽到 SSR 可一键分享到「村口」炫耀
- 「缘分骰子」掷出6点可分享给好友赠送一次机会
- 每周生成「缘分周报」：本周盲盒次数、SSR 次数、最匹配时段

### 7.2 🔮 「八字合缘」— 传统命理匹配

**设计理念**：迎合中国传统文化中对「八字合婚」的关注，为用户提供一个**趣味性+文化感**的匹配维度。定位为「参考娱乐」，不做严肃命理断言。

#### 功能描述

基于用户出生日期（年月日时）推算八字，匹配时展示双方的八字合缘指数。

#### 详细设计

```
入口：个人主页 → 「我的八字」/ 匹配卡片 → 「查看合缘」

┌─────────────────────────────────────────┐
│           🔮 八字合缘                   │
│                                         │
│   ┌──────────┐    ┌──────────┐         │
│   │   我      │    │    TA     │         │
│   │ 甲子年    │    │  乙丑年   │         │
│   │ 丙寅月    │    │  丁卯月   │         │
│   │ 戊辰日    │    │  己巳日   │         │
│   │ 庚午时    │    │  辛未时   │         │
│   └──────────┘    └──────────┘         │
│                                         │
│         💕 合缘指数：85 分              │
│         ████████████░░░░                │
│                                         │
│   📜 合婚简评：                         │
│   天干相合，地支三合，五行互补，        │
│   缘分颇深。建议多了解彼此性格，        │
│   有较好的婚配基础。                    │
│                                         │
│   ⚠️ 以上内容仅供娱乐参考              │
└─────────────────────────────────────────┘
```

#### 功能要点

| 功能点 | 说明 |
|--------|------|
| **八字录入** | 用户可选填出生时辰（年/月/日/时），非必填 |
| **隐私保护** | 出生日期不公开展示，仅用于合缘计算 |
| **合缘展示** | 匹配卡片上显示合缘指数（0-100分），点击可查看详情 |
| **合缘筛选** | 高级筛选可设置「合缘指数 ≥ 80」优先展示 |
| **免责声明** | 所有页面明确标注「仅供娱乐参考」 |

#### 技术实现思路

- 使用万年历算法推算天干地支（年柱、月柱、日柱、时柱）
- 合缘算法参考传统八字合婚规则（天干五合、地支六合/三合/六冲、五行生克）
- 综合打分：年柱权重 20%、月柱 25%、日柱 35%、时柱 20%

#### 付费设计

| 功能 | 免费 | VIP |
|------|------|-----|
| 录入八字 | ✅ | ✅ |
| 查看合缘指数 | 每日3次 | 不限 |
| 详细合缘报告 | ❌ | ✅ |
| 合缘筛选排序 | ❌ | ✅ |

### 7.3 📍 「同乡近邻」— 多维度地理匹配

**设计理念**：利用地理位置的多层维度（家乡、工作、居住），帮助用户找到「有共同地域背景」或「物理距离近」的匹配对象，增加真实见面可能性。

#### 功能描述

在现有城市筛选基础上，新增三个地理维度：

| 维度 | 说明 | 数据来源 |
|------|------|---------|
| **同城老乡** | 按籍贯/家乡城市匹配 | 用户填写家乡 |
| **工作区域** | 按工作地点商圈/园区匹配 | 用户填写工作区域 |
| **住址区域** | 按居住地片区匹配 | 用户填写住址区域（非精确地址） |

#### 详细设计

```
入口：筛选页 → 「同乡近邻」

┌───────────────────────────────────────────┐
│         📍 同乡近邻                       │
│                                           │
│  筛选维度（可多选）：                     │
│  ┌──────────────────────────────────┐     │
│  │ ☑️ 同城老乡：湖北 武汉          │     │
│  │    （与您家乡相同的人）          │     │
│  │                                   │     │
│  │ ☑️ 工作区域：北京 海淀区 中关村  │     │
│  │    （在附近上班的人）            │     │
│  │                                   │     │
│  │ ☐️ 住址区域：北京 朝阳区 望京   │     │
│  │    （住在附近的人）              │     │

│                                           │
│  排序方式：                               │
│  ○ 综合推荐   ○ 距离最近   ○ 最新活跃   │

│                                           │
│  [ 开始筛选 ]                             │
```

#### 隐私保护设计（关键）

| 策略 | 说明 |
|------|------|
| **区域模糊化** | 工作/住址仅展示到商圈/街道级别（如「中关村」「望京」），不显示精确地址 |
| **距离模糊化** | 距离显示「<1km」「1-3km」「3-5km」「5-10km」区间，不显示精确距离 |
| **用户可控** | 每个维度可单独开关（选择是否展示/是否参与匹配） |
| **非强制** | 家乡、工作区域、住址区域均为选填项 |

#### 付费设计

| 功能 | 免费 | VIP |
|------|------|-----|
| 单维度筛选 | ✅ | ✅ |
| 多维度组合筛选 | ❌ | ✅ |
| 距离排序 | ❌ | ✅ |
| 查看精确距离 | ❌ | ✅ |
| 同乡/近邻标签展示 | ✅ | ✅ |

#### 技术实现思路

- 使用高德地图/腾讯地图 API 进行地理编码和距离计算
- 用户填写区域 → 后端转换为经纬度 → 计算用户间距离
- 距离排序时使用空间索引（Geohash / R-tree）优化查询性能


## 八、商业模式

1. **会员订阅**：核心收入来源，按月/季/年付费
2. **增值道具**：超级喜欢、超级消息、超级曝光等单次付费
3. **线下活动**：付费参与线下交友活动
4. **广告/品牌合作**：婚恋产业链延伸（婚礼策划、钻戒等）
5. **🆕 游戏化收入**：缘分盲盒、缘分骰子、缘分扭蛋等运气匹配付费
6. **🆕 八字增值服务**：详细合缘报告、合缘筛选排序等 VIP 权益
7. **🆕 地理增值服务**：多维组合筛选、距离排序、精确距离查看等 VIP 权益

## 十二、技术选型与架构 🔧

> ⚡ 2026-06-09 更新：小程序框架由 uniapp 切换为 **Flutter**

### 12.0 技术选型总览

| 层级 | 技术 | 选型说明 |
|------|------|---------|
| **小程序端** | **Flutter** (Dart) | 高性能渲染引擎、一套代码双端运行、丰富动画效果（缘分盲盒/骰子/扭蛋动效天然适配）、成熟的微信小程序支持 |
| **后端框架** | Java **Spring Boot** | 渐进式模块化单体，按功能分包（user/karma/bazi/geo/chat/village），后期可拆微服务 |
| **数据库** | **MySQL** 8.0 | 主数据存储，支持 JSON 类型、空间索引 |
| **缓存** | **Redis** | 用户 session、缘分每日限制、热点数据缓存、八字合缘结果缓存 |
| **搜索引擎** | **Elasticsearch** | 用户搜索、匹配推荐、区域搜索 |
| **消息推送** | WebSocket | 即时通讯、匹配通知 |
| **云服务** | 阿里云 ECS | 应用部署、文件存储 (OSS)、CDN 加速 |

### 12.0.1 为什么选择 Flutter？

| 维度 | Flutter | uniapp |
|------|---------|--------|
| **渲染性能** | ⭐⭐⭐⭐⭐ Skia 引擎原生渲染，60fps 动画丝滑 | ⭐⭐⭐ WebView/Native 混合渲染 |

### 12.4 核心算法

#### 12.4.1 八字合缘算法（Python 参考实现）

```python
# 天干五合映射
HEAVENLY_STEM_COMBINATION = {
    ('甲', '己'): 100, ('乙', '庚'): 100, ('丙', '辛'): 100,
    ('丁', '壬'): 100, ('戊', '癸'): 100
}

# 地支六合映射
EARTHLY_BRANCH_COMBINATION = {
    ('子', '丑'): 100, ('寅', '亥'): 100, ('卯', '戌'): 100,
    ('辰', '酉'): 100, ('巳', '申'): 100, ('午', '未'): 100
}

# 地支三合
EARTHLY_BRANCH_TRIPLE = {
    '申子辰': 90, '巳酉丑': 90, '寅午戌': 90, '亥卯未': 90
}

# 地支六冲（负面）
EARTHLY_BRANCH_CLASH = {
    ('子', '午'): -60, ('丑', '未'): -40, ('寅', '申'): -40,
    ('卯', '酉'): -40, ('辰', '戌'): -40, ('巳', '亥'): -40
}

# 五行相生
FIVE_ELEMENT_GENERATE = {
    '木': '火', '火': '土', '土': '金', '金': '水', '水': '木'
}

def calculate_bazi_match(bazi_a, bazi_b):
    """计算八字合缘指数"""
    scores = {}

    # 年柱 (权重 20%)
    scores['year'] = _pillar_score(bazi_a['year'], bazi_b['year'],
                                    HEAVENLY_STEM_COMBINATION,
                                    EARTHLY_BRANCH_COMBINATION,
                                    EARTHLY_BRANCH_CLASH)

    # 月柱 (权重 25%)
    scores['month'] = _pillar_score(bazi_a['month'], bazi_b['month'], ...)

    # 日柱 (权重 35%) — 最重要
    scores['day'] = _pillar_score(bazi_a['day'], bazi_b['day'], ...)

    # 时柱 (权重 20%)
    scores['hour'] = _pillar_score(bazi_a['hour'], bazi_b['hour'], ...)

    # 五行互补加分（额外 ±10 分）
    element_bonus = _five_element_bonus(bazi_a['elements'], bazi_b['elements'])

    total = (scores['year'] * 0.20 +
             scores['month'] * 0.25 +
             scores['day'] * 0.35 +
             scores['hour'] * 0.20 +
             element_bonus)

    return max(0, min(100, int(total)))
```

#### 12.4.2 缘分盲盒匹配算法

```
function selectBlindBoxUser(userId):
    1. 获取用户偏好（性别、年龄范围、学历偏好）
    2. 从候选池中随机抽取：
       - 排除已匹配过的用户（当日内）
       - 排除用户已 skip 的用户（7日内）
       - 保证同城用户占比 ≥ 50%
    3. 返回随机选中的用户

function selectDiceUser(userId, dicePoint):
    switch dicePoint:
        case 1, 2: → 从全量候选池随机抽
        case 3, 4: → 仅从同城候选池随机抽
        case 5:    → 取匹配度 top 20 中随机抽 1 个
        case 6:    → 不匹配，直接奖励 super_like
```

#### 12.4.3 扭蛋 SSR 概率实现

```python
import random

def draw_gacha():
    """单次扭蛋抽卡"""
    rand = random.random() * 100

    # 保底机制：如果连续 30 抽无 SSR，第 31 抽强制 SSR
    if consecutive_without_ssr >= 30:
        consecutive_without_ssr = 0
        return 'SSR'

    if rand < 3:           # 3% SSR
        consecutive_without_ssr = 0
        return 'SSR'
    elif rand < 15:        # 12% SR
        return 'SR'
    elif rand < 40:        # 25% R
        return 'R'
    else:                  # 60% N
        consecutive_without_ssr += 1
        return 'N'
```

以下从 99 条 UX 指南中筛选出与青藤之恋最相关的实践：

#### 新用户引导

| ✅ 必须做 | ❌ 不要做 |

#### 社交功能

| ✅ 必须做 | ❌ 不要做 |
|-----------|-----------|
| 触控目标 ≥ 44×44pt（滑动/喜欢按钮） | 小图标无放大触控区域 |
| 按压反馈 80-150ms（喜欢/匹配动画） | 点击无视觉响应 |
| 模态框提供明确的关闭/撤销按钮 | 模态框无退出路径 |
| 手势交互后备：滑动同时提供按钮操作 | 仅依赖手势（可访问性问题） |

#### 信任与安全

| ✅ 必须做 | ❌ 不要做 |
|-----------|-----------|
| 颜色不是唯一指示器（匹配/错误状态加图标+文字） | 仅靠红绿色传达信息 |
| 破坏性操作（解除匹配）二次确认 | 一键解除无确认 |
| 错误信息说明原因 + 恢复路径 | 「操作失败」无进一步引导 |

### 16.8 📐 核心页面设计规范

#### 底部导航栏（5 项原则）

```
┌──────┬──────┬────────┬──────┬──────┐
│ 寻觅  │ 村口  │  匹配   │ 消息  │ 我的  │
│ 🔍   │ 🏘️   │   💕   │ 💬   │ 👤   │
└──────┴──────┴────────┴──────┴──────┘
```

- 最多 5 项、每项图标+文字标签
- 当前页高亮（品牌色 + 加粗）
- 消息项有未读徽章（红点）
- 不嵌套子导航于底部栏
- 大屏（≥1024px）切换为侧边栏

#### 卡片设计规范

```
卡片组件 Token：
  bg: var(--color-card)          // #FFFFFF

  radius: 16px                   // 大圆角（友好感）

  padding: 16px
  gap: 12px（卡片间距）

卡片悬停态：
  transform: translateY(-2px)    // 微上浮
  shadow: 加深
  transition: 200ms ease-out
```
